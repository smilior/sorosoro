01 要件定義 · 04/04

非機能要件

おうち掃除ログ（sorosoro）の品質目標・レスポンシブ方針・認証認可・法務要件を定義する。

確定メモ

MVP は Next.js + Vercel + Turso + Better Auth + Google OAuth を前提とする品質目標。会員制・ユーザー単位データ隔離が必須。

モバイルファースト。ソフトな UX（責めない色・タップ ≥ 44px・メモ帳トーン）を品質の一部として扱う。

品質目標値

区分 項目 目標値 前提条件
性能主要画面の表示初回・再訪とも 2 秒以内（中位帯スマホ・通常回線）Vercel エッジ配信 + Server Components / Server Actions。モバイルファースト
性能1 タップ記録の体感操作完了から UI 反映は楽観更新を含め素早く。サーバー確定は通常回線で数 100ms 台を目標Server Actions + Turso。失敗時はトーストで通知し再試行可能
性能想定同時接続数個人向け小規模（同時数十ユーザー程度を快適に扱えること）Hobby / Free 枠での個人アプリ規模。大規模 B2B 負荷は対象外
性能データ増加見込みユーザーあたりタスク数十件・ログ数百件規模を快適に操作Turso（libSQL）。クエリは user_id で絞る
可用性稼働率(SLO)Vercel + Turso の無料 / Hobby 枠の現実的稼働を目安（目標 99% 前後）。計画メンテは事前告知マネージド基盤の SLA に依存。個人アプリとして過剰な冗長化はしない
可用性復旧目標 RTO / RPORTO: 再デプロイ・設定復旧で数分〜数時間 / RPO: Turso バックアップ方針に準拠（日次相当を目標）ユーザーデータは Turso。コードは Git 管理
セキュリティ準拠基準HTTPS 必須。Better Auth のセッション Cookie。Google OAuth。XSS 防止（ユーザー入力の適切なエスケープ）。データは user_id で隔離他ユーザーの tasks / task_logs にアクセスできないこと
セキュリティ脆弱性診断公開前に依存パッケージ監査と認証・認可・XSS 観点の自己チェックを実施課金なし。個人アカウントのメール等を扱うため最低限の点検は必須
運用保守ログ保管期間Vercel のランタイムログはプラットフォーム既定に従う。個人を特定しうるログは最小限サーバーレス / マネージド
運用保守バックアップ頻度Turso 側のバックアップ・スナップショット機能を利用（利用可能プランに合わせる）ユーザーデータの正はクラウド DB
移行性既存データ移行無（新規アプリ）。スキーマ変更は Drizzle マイグレーションで管理本番 DB は Turso リモート
UXソフトな見た目・操作警告アイコン・強い赤禁止。タップ領域 ≥ 44px。記録に確認ダイアログを挟まない企画書デザイン原則・トークンに準拠

レスポンシブ対応方針

デバイス 画面幅 対応
スマートフォン320px - 767px必須（プライマリ。モバイルファースト）
タブレット768px - 1023px対応（レイアウトを崩さず利用可能）
デスクトップPC1024px 以上対応（コンテンツをセンタリングし最大幅を制限）

認証・認可要件

項目 内容
会員制要（ログイン必須で自分のデータにアクセス）
認証方式Better Auth + Google OAuth のみ（MVP）。メール / パスワード登録は提供しない
OAuth 資格情報Google Cloud Console（GCP）でクライアント ID / シークレットを管理
セッション管理Better Auth のセッション Cookie。サーバー側セッション情報は Turso に保存
データ隔離すべてのタスク・記録は user_id で所有。他ユーザーのデータは参照・更新不可
ロール 想定利用者 主な権限
ログインユーザーGoogle アカウントでサインインした本人自分のタスク・履歴の参照・登録・編集・削除。ログアウト

項目 要否 備考
