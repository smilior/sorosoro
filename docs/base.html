本資料について 本資料は、労働安全衛生システム（ap-safedx／災害水平展開アプリ）の「災害水平展開」機能に組み込む「AI検索・分析機能」のビジネス価値とMVP検証計画を整理したドキュメントです。 本機能の目的・価値提案・成功基準を明確にし、ステークホルダー間の認識を揃えます。

※ 本機能は既存アプリ（ap-safedx）の「災害水平展開」機能への機能追加であり、既存の技術基盤・認証基盤・組織/権限モデル、およびBOX連携を共用します。

プロジェクト名：労働安全衛生システム（ap-safedx） ／ AI検索・分析機能

作成日: 2026-07-17 バージョン: 1.0 作成者: DX推進グループ

0. 背景（なぜ本機能を作るのか）

0.1 災害水平展開と3つの災害区分

労働安全衛生システム（ap-safedx）では、社内外で発生した災害情報を全社員へ周知し、再発防止に活かす「災害水平展開」を運用している。災害は以下の3区分で登録・回覧される。

区分 内容 主な様式
ヒヤリ・ハット報告 実際に体感した／潜在的に想定したヒヤリ・ハット。原因・リスク評価・所属長／安全事務局コメントを含む ヒヤリ・ハット報告書
救急箱災害 救急箱で処置した程度の労働災害。災害の型・被災者情報・災害状況の概要・水平展開・安全事務局コメントを含む 救急箱災害報告書（様式－４保安）
災害速報（重大災害） 休業・不休業を伴う労働災害の速報。労働災害報告書として被災状況・原因・対策を含む 労働災害報告書（様式－２保安）
  • 各報告書はPDF等のファイルとして作成され、BOXに保管される。災害情報の登録時にはBOXの共有リンクが紐づけられ、回覧対象者はそのリンクから資料を参照する。

0.2 現状の課題

項目 現状
過去災害の活用 過去に蓄積された多数の災害報告書（ヒヤリ・ハット／救急箱／災害速報）が、区分・年度・拠点ごとに散在し、横断的に探せない
類似災害の検索 「過去に似た事故はあったか」を調べるには、担当者が記憶と手作業でファイルを漁るしかなく、見落とし・属人化が発生する
検索手段 ファイル名・区分での絞り込みが中心で、本文（作業内容・原因・対策）を対象にした検索ができない
気付きの提供 回覧された報告書は、読み手が全文を読んで初めて要点が分かる。要点・気付きが即座に伝わらず、確認が形骸化しやすい
傾向把握 災害の型・原因・拠点別の傾向を把握するには、都度Excelで集計する必要があり、タイムリーな傾向分析ができない

位置づけ: 26年度テーマ候補のうち、災害水平展開の高度化施策。No.1（議事録回覧の効率化）に続く優先施策として、蓄積データのAI活用を図る。

0.3 本機能で実現すること

災害情報の登録（回覧）時に、BOX共有リンクから報告書ファイルを取得し、以下を自動で行う。

  1. RAGへの自動登録：報告書の本文をベクトル化してRAGに登録し、あいまい検索（自然言語検索）を可能にする
  2. 要約（気付き）の自動生成：報告書を要約し、読み手に気付きを与える文章を生成して記事確認画面に表示する

蓄積されたデータに対しては、以下のAI活用を提供する。

  1. 自然言語検索：全社員が「過去に似た事故はあるか」等を自然文で検索し、根拠（出典）付きで類似災害・再発防止策を引き出す
  2. 傾向分析・レポート：災害の型・原因・拠点・区分などの観点で傾向を分析し、レポートを生成する

1. 顧客セグメント（Customer Segments）

主要顧客

  • 全社員（検索・閲覧者）：自分の作業や職場に関連する過去災害・類似災害・再発防止策を知りたい全ての従業員。主な確認者は三井金属総合研究所の研究者で、工場系の災害を実験・試作の安全に活かす。回覧された報告書の要約（気付き）を読み、必要に応じて自然言語で検索する
  • 安全衛生管理者（組織管理者）＝災害情報の登録者：災害情報を登録・回覧する担当者。登録時にRAG登録・要約生成が自動実行される
  • 安全事務局・安全衛生スタッフ（分析者）：災害の型・原因・拠点別の傾向を把握し、教育・対策立案に活かす担当者

顧客の特徴

  • 全社員が対象だが、確認の中心は三井金属総合研究所の研究者。研究者からスタッフ職まで、ITリテラシーの幅が広い
  • 検索者は「専門的な検索構文」ではなく、日常の言葉で問いたい（自然言語検索がMUST）
  • 分析者は、区分・年度・拠点をまたいだ横断的な傾向を、集計作業なしで把握したい

2. 価値提案（Value Propositions）

全社員（検索・閲覧者）向け

  • 自然言語での類似災害検索：「フォークリフトで転落しそうになった事例」のように日常の言葉で過去災害を検索できる
  • 根拠付きの回答：検索結果には出典（どの報告書か）が明示され、元資料（BOX）を確認できる
  • 要約（気付き）で要点把握：回覧された報告書の要点と再発防止の気付きが、記事確認画面で一目で分かる

安全衛生管理者（登録者）向け

  • 登録するだけでAI活用：災害情報を登録・回覧すると、RAG登録と要約生成が自動で行われる（追加作業なし）
  • 気付きの自動付与：読み手に伝わりやすい要約が自動生成され、周知の質が上がる

安全事務局・分析者向け

  • 傾向分析の自動化：災害の型・原因・拠点・区分などの傾向を、集計作業なしでレポート化できる
  • 横断的な把握：区分・年度・拠点をまたいで蓄積データを横断分析できる

全体的な価値（安全文化・再発防止）

  • 再発防止の高度化：過去の類似災害・対策を素早く引き出し、同種災害の再発を防ぐ
  • 周知の実効性向上：要約（気付き）により、回覧された災害情報が「読まれる・伝わる」ものになる
  • 属人化の解消：過去災害の検索を担当者の記憶に頼らず、誰でも同じ品質で行える
  • 既存基盤の活用：災害水平展開・BOX・認証基盤をそのまま活用し、投資を最小化する

提供方法

7. コスト構造（Cost Structure）

初期コスト

  • 開発費用：AI検索・分析機能の開発工数（既存基盤流用のため新規インフラ投資は最小）
  • 過去分の一括取り込み費用：蓄積済み報告書のRAG登録（埋め込み）にかかるAPI費用

運用コスト

  • 既存インフラ費用に内包：Azure VM、PostgreSQL、Redis は既存アプリと共用
  • Azure OpenAI 従量課金：埋め込み・要約・検索応答生成のトークン費用（新規に発生）
  • ベクトルストア費用：既存DBに内包、または軽量な追加コンポーネント

コスト最適化方針

  • 既存基盤の最大活用：認証・組織/権限・BOX・災害水平展開を流用
  • モデルの使い分け：高精度が必要な用途は上位モデル、要約等は軽量モデル（5.4-mini）でコスト最適化
  • 登録時の増分処理：新規登録分のみを都度RAG登録し、無駄な再計算を避ける

8. 検証したい仮説（Hypothesis）

仮説1: 自然言語検索の受容性・実用性

「全社員は、区分・年度での絞り込みより、自然言語での類似災害検索を好み、実務で使う」 - 背景: 現状は本文検索ができず、過去災害の調査が属人化している - 検証ポイント: 日常の言葉で問うて、期待する類似災害・対策が根拠付きで返るか - リスク: 報告書のフォーマット揺れ・OCR品質により検索精度が上がらない可能性

仮説2: 要約（気付き）の効果

「登録時に自動生成される要約は、回覧された災害情報の要点把握と気付きに役立つ」 - 背景: 全文を読まないと要点が伝わらず、確認が形骸化しやすい - 検証ポイント: 要約が正確で、読み手に再発防止の気付きを与えられるか - リスク: 要約の誤り・重要点の欠落（ハルシネーション）による誤解

仮説3: RAG自動登録の運用性

「災害情報の登録（回覧）時に、BOX共有リンクから報告書を取得し自動でRAG登録できる」 - 背景: 報告書はBOXに保管され、災害情報にBOXリンクが紐づく - 検証ポイント: 各様式（ヒヤリ・ハット／救急箱／災害速報）のPDFを安定して取得・テキスト化・登録できるか - リスク: 画像PDF・レイアウト崩れ・BOX権限によりテキスト抽出に失敗する可能性

仮説4: 傾向分析・レポートの価値

「災害の型・原因・拠点・区分などの傾向を、集計作業なしでレポート化でき、対策立案に役立つ」 - 背景: 傾向把握は都度のExcel集計に依存している - 検証ポイント: 分析者が求める観点でタイムリーにレポートが得られるか - リスク: 分類・タグの粒度が揃わず、傾向の信頼性が低下する可能性

ペルソナ設定

対象ユーザーの人物像

本資料について 本資料は、労働安全衛生システム（ap-safedx）の「災害水平展開」機能に組み込む「AI検索・分析機能」の対象ユーザーを具体的な人物像として定義したドキュメントです。 ユーザーの課題・ニーズ・ゴールを明確にし、ユーザー中心の設計・開発を実現します。

  • 年齢：34歳
  • 所属：三井金属総合研究所 機能材料研究部

  • 勤務形態：日勤（実験・試作あり）
  • ITリテラシー：中〜高程度（PC・文献検索は日常的。ただし安全システムの検索構文には不慣れ）

背景・状況

  • 実験室や試作ライン（パイロット設備）で、金属材料・薬品・高温炉・回転機器などを扱う
  • グループの工場等で発生した災害情報が回覧されてくる。多くは工場系（製錬・機械・薬品取扱い等）だが、実験・試作で扱う設備・薬品・作業に共通するリスクを含む
  • 自分の研究テーマに関係する事例だけを知りたいが、報告書の全文を読む時間が取りにくい
  • 「区分」「年度」で絞る検索はあるが、自分の実験作業に似た事例を言葉で探す手段がない

困りごと・ニーズ

困りごと

  • 工場事例と自分の研究が結び付けにくい：工場系の災害が、実験・試作のどの作業に効くのか判断しにくい
  • 似た事例が探せない：薬品取扱い・加熱・回転体・運搬など、自分の作業に近い事例を日常の言葉で探せない
  • 要点が分からない：回覧された報告書の要点・気付きが即座に伝わらない

ニーズ

  • 自然言語で聞きたい：「◯◯を加熱する実験に近いヒヤリはあるか？」のように問いたい
  • 要点だけ知りたい：報告書の要点と再発防止の気付きを短時間で把握したい
  • 根拠を確認したい：回答の出典（元の報告書）を確認できると安心

ゴール

4. 制約事項・前提条件

制約事項

  • Entra ID SSO・組織／部署／ユーザー情報・権限は既存基盤（ap-safedx／ap-auth）を利用（新規構築不要）
  • 新規権限の追加は行わない（既存ロールで対応）
  • RAG登録の対象は3区分（ヒヤリ・ハット／救急箱／災害速報）の報告書。議事録等の横断はPhase 2
  • 報告書ファイルはBOXに保管され、災害情報のBOX共有リンク経由で取得する
  • AI連携先は Azure OpenAI（埋め込み：text-embedding-3-large、生成：GPT-5.4／5.4-mini を検討）
  • 被災者名等はマスキング運用を前提とする

前提条件

  • 既存の ap-safedx／ap-auth／インフラ（PostgreSQL・Redis・スケジューラ）が稼働している
  • BOXの共有リンクから報告書ファイルを取得できる権限・APIが利用可能
  • Azure OpenAI の利用環境（エンドポイント・モデルデプロイ・キー）が用意されている
  • ベクトル検索基盤（PostgreSQL拡張または専用ストア）が利用可能

5. 用語

用語 説明
RAG Retrieval-Augmented Generation。検索で集めた根拠（報告書の該当箇所）を基に、AIが回答を生成する方式。出典を提示でき、根拠のない生成を抑えられる
埋め込み（Embedding） 文章を意味的な数値ベクトルに変換すること。意味の近さで検索（あいまい検索）できるようになる
チャンク化 報告書本文を検索に適した一定の単位に分割すること。各チャンクに区分・年度・拠点等のメタデータを付与する
ベクトルストア 埋め込みベクトルを保存し、類似度で高速に検索するための基盤（RAGインデックス）
OCR 画像化されたPDF等から文字を読み取り、テキスト化する技術
ハルシネーション AIが根拠なく事実と異なる内容を生成すること。出典提示・「該当なし」明示で抑制する
